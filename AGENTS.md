# AGENTS.md — contributor guide for coding agents working on Limux itself

> **Note:** This file is for agents (Codex / Claude Code / OpenCode / Gemini /
> Aider / etc.) **editing the limux source tree**. A *different* `AGENTS.md`
> is auto-generated by `limux agent-team` into the user's shared cwd at
> runtime — that one documents the inter-agent message protocol, not this
> repo's conventions. Don't confuse them.

## Repo in one paragraph

Limux is a GTK4 + libadwaita + libghostty terminal workspace manager for
Linux, ported from manaflow-ai's macOS `cmux`. The UI is native GTK; the
terminal rendering is embedded `libghostty.so`. The app exposes a Unix
socket control API so a `limux-cli` binary (and any script or coding
agent) can drive workspaces, panes, surfaces, and notifications from
outside the GUI.

## Crate map (`rust/`)

| Crate | Role |
|---|---|
| `limux-protocol` | JSON command envelope types (V2 request/response). |
| `limux-core` | In-process control state. Workspaces / surfaces / notifications. **IDs are `u64` here.** |
| `limux-control` | Unix socket server + `socket_path::resolve_socket_path(None, SocketMode::Runtime)`. |
| `limux-ghostty-sys` | Raw FFI bindings to `libghostty.so` (see `ghostty/include/ghostty.h`). |
| `limux-host-linux` | The GTK binary. Window / sidebar / panes / tabs / terminal + `control_bridge.rs` that forwards socket commands into the GTK main loop. |
| `limux-cli` | The `limux` CLI binary (agent-facing surface). |

## Two binaries that both call themselves `limux` (gotcha!)

1. **`target/debug/limux`** — the GTK *app* from `limux-host-linux`.
   Only understands GTK flags (`--help-gapplication`, `--help-gtk`, etc.)
   and will **not** respond to CLI subcommands like `agent-team` or
   `identify`.
2. **`target/debug/limux-cli`** — the actual CLI from `limux-cli`. This
   is what `agent-team`, `notify`, `claude-hook`, etc. live on.

Installed users get `limux` as a symlink to the CLI binary, but when you
build locally, use `./target/debug/limux-cli …` explicitly.

```bash
cargo build -p limux-cli --bin limux-cli
./target/debug/limux-cli --help
```

## Quality gate

The single canonical check:

```bash
./scripts/check.sh
# cargo fmt --check
# cargo clippy --workspace --all-targets -- -D warnings
# cargo test --workspace     (184+ unit tests)
```

Keep `main` green under this script. Run `cargo fmt` before every commit
— rustfmt will otherwise complain about multi-line `push((…))` calls
that initial patches often produce.

For faster iteration, prefer narrowly-scoped checks:

```bash
cargo check -p limux-host-linux     # GTK/FFI feedback in ~5s
cargo test  -p limux-cli            # CLI unit tests only
cargo check --workspace             # Full feedback without building binaries
```

## Headless integration testing (no GUI, no display)

The full GTK app can run under `xvfb-run` with zero display hardware —
useful for CI, and for agent-run smoke tests. System deps:

```bash
# Arch-family
sudo pacman -S --needed xorg-server-xvfb
# Debian/Ubuntu
sudo apt-get install -y xvfb
```

Smoke pattern:

```bash
(cd ghostty && zig build -Dapp-runtime=none -Doptimize=ReleaseFast)
cargo build --release -p limux-host-linux -p limux-cli

xvfb-run -a ./target/release/limux &
HOST_PID=$!

# Wait for the Unix socket to come up, then drive it with limux-cli
./target/release/limux-cli list-workspaces
./target/release/limux-cli agent-team --agents codex,claude --cwd /tmp/demo
./target/release/limux-cli send --workspace claude '<agent-msg …>'
./target/release/limux-cli notify --workspace claude 'Hi'

kill $HOST_PID
```

`limux-cli agent-team --dry-run` additionally works with **no host
running** — it short-circuits socket calls and just exercises the
launcher map + `build_agents_md` path, which is what the
`agent_team_tests` unit tests cover.

### Control bridge (`rust/limux-host-linux/src/control_bridge.rs`)

There are **two** control servers in the workspace:

1. Standalone `limux-control-server` binary — uses
   `limux_core::Dispatcher` + `ControlState` and supports the **full**
   command vocabulary. Used in tests and when the GUI is not running.
2. **Embedded bridge inside `limux-host-linux`** — only routes a narrow
   subset of methods to the GTK main loop.

When the GUI is running, the CLI targets the bridge via the runtime
socket. Commands that aren't routed by the bridge return errors — this
is precisely why Phase 2 (full proxy) is the remaining critical path for
richer agent-to-agent flows (read-screen across peers, cross-surface
key injection, etc.).

### ID mismatch — critical

- **GTK layer** (`window.rs`, `pane.rs`): workspace id is `String`,
  pane id is `u32`, tab id is a uuid `String`. Surface id string format
  is `"<pane_id>:<tab_id>"` (see `TabDragPayload` in `pane.rs`).
- **Control core** (`limux-core/src/lib.rs`): workspace/surface ids are
  `u64`, encoded externally via `encode_handle_id()`.
- When building the env var for a shell, construct
  `LIMUX_SURFACE_ID = format!("{pane_id}:{tab_id}")`. Do **not** try to
  use a `SurfaceId` type in host-linux — none exists there (the
  similarly-named one lives in `limux-cli`).

### `PaneCallbacks` pattern

Panes don't know their owning workspace directly. `PaneCallbacks` in
`pane.rs` holds closures captured at pane-creation time in `window.rs`
(function around line 3173). When you need the workspace id inside the
pane layer, add a callback to `PaneCallbacks` and supply it from
`window.rs` by capturing `ws_id.to_string()` near the existing
`let ws_id_for_* = …` lines. **There is exactly one call site that
constructs `PaneCallbacks`** — updating it is the only wiring you need.

### Ghostty FFI — `env_vars` lifetime

`ghostty_surface_config_s` has `env_vars: *mut ghostty_env_var_s` +
`env_var_count`. Ghostty `dupeZ`s both key and value into its own config
arena (`ghostty/src/apprt/embedded.zig:573`), so the pointers only need
to survive the single `ghostty_surface_new` call. Safe pattern in
`terminal.rs::create_terminal`:

1. Build a `Vec<CString>` of keys and values.
2. Build a `Vec<ghostty_env_var_s>` with raw pointers into them.
3. Stamp onto the config, call `ghostty_surface_new`, let everything drop.

No static lifetime, no leaks. Re-realize on reparenting is guarded by
the early-return at `if let Some(surface) = *surface_cell.borrow()`, so
env vars only apply on first realize — fine, because the pty survives
reparent.

## Agent integrations — the user-facing CLI surface

```bash
limux notify [--workspace <id|name>] [--subtitle <…>] [--body <…>] <title>
limux claude-hook | opencode-hook | gemini-hook \
        --event <name> [--subtitle <…>] [--body <…>] [--title <…>]
limux agent-team [--agents codex,claude[,opencode,gemini]] \
        [--cwd <path>] [--no-launch] [--dry-run]
```

Extensibility hooks for adding more agents:

- `agent_launch_command` in `rust/limux-cli/src/main.rs:922` maps each
  agent name to `(cli_binary, launch_arg_string)`. Add a tuple, done.
- `build_agents_md` in `rust/limux-cli/src/main.rs:1051` takes
  `&[(agent, workspace_name, workspace_id, launch_cmd)]` — preserve the
  tuple shape when extending; `agent_team_tests` covers the template.
- Both `surface.send_text` and `notification.create` in
  `control_bridge.rs` pass `allow_name=true` to
  `parse_optional_workspace_target`, which is what makes `--workspace
  claude` work without a UUID. When adding a new bridge route that an
  agent might call against a peer, do the same.

## Env contract

Every terminal limux spawns inherits:

- `LIMUX_WORKSPACE_ID` — the current workspace's UUID
- `LIMUX_SURFACE_ID` — `"<pane_id>:<tab_id>"`
- `LIMUX_PANE_ID`, `LIMUX_TAB_ID`
- `LIMUX_SOCKET` — the control socket path

The CLI reads these with `env::var(...).ok()` fallbacks after
`parse_opt(args, "--workspace")`. Mirror that pattern in any new
subcommand you add.

## Conventions

- `main` tracks the upstream remote. Active work is on topic branches
  (e.g. `feat/cmux-parity`, `fix/…`). The agent does not open issues or
  PRs for this repo directly.
- Commit identity in use: `am-will <william@am-will.dev>`.
- The vendored `ghostty/` tree is **read-only** from the limux-layer
  perspective. Everything goes through the C API in
  `ghostty/include/ghostty.h`.
- Prefer small domain modules over monolithic files; move test modules
  out of huge production files when they obscure the main codepath
  (see `docs/maintainability.md`).
- Treat clippy findings as maintainability work, not optional cleanup.
